Spring Web

Servlet Lifecycle

A Servlet lifecycle is managed by the container in which it has been deployed.

  • https://docs.oracle.com/javaee/6/tutorial/doc/bnafi.html
  • http://docs.jboss.org/jbossas/javadoc/7.1.2.Final/javax/servlet/Servlet.html

ContextLoaderListener

This is the bootstrap listener to manipulate the Spring’s root WebApplicationContext (being shared by all servlets within the web application). Behind the scene, it ties the lifecycle of the ApplicationContext to the lifecycle of the ServletContext. For normal usage scenarios, ContextLoaderListener might be not necessary.

<web-app id="..."
  version="3.0"
  xmlns="http://java.sun.com/xml/ns/javaee"
  xmlns:xsi="http://www.w3.org/2001/XMLSchema-instance"
  xsi:schemaLocation="http://java.sun.com/xml/ns/javaee http://java.sun.com/xml/ns/javaee/web-app_3_0.xsd">
	
	<listener>
		<listener-class>org.springframework.web.context.ContextLoaderListener</listener-class>
	</listener>
  <context-param>
    <param-name>contextConfigLocation</param-name>
    <param-value>...,...</param-value>
  </context-param>
</web-app>

Since Servlet 2.4, Servlet context listeners must execute immediately after the Servlet context for the web application is created and availalble to serve the first request. That’s why ContextLoaderListener is ideal and often used to initialise Spring’s WebApplicationContext.

When starting, ContextLoaderListener will inspect the contextConfigLocation parameter within <context-param>...</context-param> to retrieve the XML configuration file. If the parameter does not exist, the listener will go for the default config file /WEB-INF/applicationContext.xml. Thus, when you don’t define contextConfigLocation nor don’t have applicationContext.xml, Spring will raise an exception:

org.springframework.beans.factory.BeanDefinitionStoreException: IOException parsing XML document from ServletContext resource [/WEB-INF/applicationContext.xml]; nested exception is java.io.FileNotFoundException: Could not open ServletContext resource [/WEB-INF/applicationContext.xml]

DispatcherServlet

This is the main dispatcher for HTTP request handlers that dispatch a request to the corresponding registered handlers. A web application can define several DispatcherServlets each operates on its own namespace with separate context, mappings, handlers.

Note that DispatcherServlet extends FrameworkServlet that inspects its contextClass parameter at the servlet init-param level, falling back to the default context class, XmlWebApplicationContext, if the parameter does not exist. DispatcherServlet also supports a contextConfigLocation parameter (inside servlet <init-param>...</init-param>) for setting paths to the configuration files. If not specified, the default configuration location is based on the namespace of the servlet, i.e. /WEB-INF/<servetname>-servlet.xml (with XmlWebApplicationContext).

<servlet>
  <servlet-name>mvc</servlet-name>
  <servlet-class>org.springframework.web.servlet.DispatcherServlet</servlet-class>
  <init-param>
    <param-name>contextConfigLocation</param-name>
    <param-value>/WEB-INF/my-config-servlet.xml</param-value>
  </init-param>
  <load-on-startup>1</load-on-startup>
</servlet>

public class MyWebAppInitializer extends AbstractAnnotationConfigDispatcherServletInitializer {

    @Override
    protected Class<?>[] getRootConfigClasses() {
        return new Class<?>[] { RootConfig.class };
    }

    @Override
    protected Class<?>[] getServletConfigClasses() {
        return new Class<?>[] { App1Config.class };
    }

    @Override
    protected String[] getServletMappings() {
        return new String[] { "/app1/*" };
    }
}

Web Application Initialisation

Method 1: Servlet Context Listener (web.xml)
  1. A web application WAR is being deployed by user.
  2. Servlet container (e.g. Tomcat, Jetty) reads web.xml.
  3. Servlet context listener ContextLoaderListener is being instantiated (if defined as <listener> inside web.xml) by servlet container.
    1. ContextLoaderListener creates new WebApplicationContext / XmlWebApplicationContext using XML configuration (by default via the listener’s parameter contextConfigLocation , if exists, otherwise /WEB-INF/applicationContext.xml).
    2. The ROOT context beans are registered and instantiated by BeanFactory inside the application context.
  4. DispatcherServlet is being instantiated by servlet container.
    1. DispatcherServlet creates its own WebApplicationContext (WEB-INF/{servletName}-servlet.xml by default) with the ROOT context as its parent.
    2. The servlet beans are registered and instantiated by BeanFactory inside the application context.
    3. DispatcherServlet registers some default beans in case no beans are defined.
Method 2: Servlet Container Initializer (non web.xml) (since Servlet 3.0+)
  1. A Web application WAR is being deployed by user.
  2. Servlet container searches for classes implementing ServletContainerInitializer via Java’s ServiceLoader.
    • SpringServletContainerInitializer is instantiated by the servlet container.
  3. Spring’s initializer reads Web application’s class-path and searches for WebApplicationInitializer implementations.
    • AbstractAnnotationConfigDispatcherServletInitializer provides an abstract implementation of WebApplicationInitializer with simpler methods for setting servlet mapping and configuration location.
  4. An implementation of WebApplicationInitializer is found and instantiated by SpringServletContainerInitializer.
    1. The WebApplicationInitializer creates new ROOT WebApplicationContext with XML or @Configuration based configuration.
    2. The WebApplicationInitializer creates new servlet WebApplicationContext with XML or @Configuration based configuration.
    3. The WebApplicationInitializer creates and registers new DispatcherServlet with the context from previous step.
  5. Servlet container finishes the web application initialization and instantiates components which were registered by their class in previous steps.
