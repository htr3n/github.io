Hibernate: Migrating 3.x to 5.x

Dec 5, 2018 in DEV
database orm hibernate spring-framework
4 min read

Recently I had to upgrade a codebase using Hibernate 3.x ORM on Spring 3 to Hibernate 5 (and Spring 5, respectively). Here I note down some aspects and issues that I have to deal with during this migration.

An overview of a simple data access architecture is as following. The current codebase leverages lots of Hibernate native API that are rich and optimized. I am looking forward to be less dependent on Hibernate-specific stuffs and using standards such as Java Persistence API (JPA) and Java Transaction API (JTA) as much as possible in my projects. As such, it is possible to switch the projects to another JPA provider, for instance EclipseLink, if needed.

The Legacy HibernateDaoSupport/HibernateTemplate

Spring ORM 3 provides several facades to ease data access with Hibernate. Using HibernateDaoSupport and HibernateTemplate we can be highly productive due to their rich API and can also leverage Spring transaction management. HibernateDaoSupport is indeed a wrapper of HibernateTemplate that simplifies data access code than raw Session API. In turn, HibernateTemplate actualls ties to a Hibernate’s SessionFactory behind the scene. So the basic CRUD code for accessing database is rather simple and straightforward.

public class CustomerDaoImpl extends org.springframework.orm.hibernate3.support.HibernateDaoSupport {
    public Customer create(Customer customer) {
      	Customer customer = new Customer();
      	customer.setId(...);
      	...
      	saveOrUpdate(customer);  
    }
    public Customer retrieve(String customerId) {
      	return getHibernateTemplate().get(Customer.class, customerId);
    }
    public void saveOrUpdate(Customer customer){
      	getHibernateTemplate().saveOrUpdate(customer);    
    }
    public void delete(Customer customer) {
        getHibernateTemplate().delete(customer);
    }
}

Nevertheless, since Hibernate 3.0.1, we can leverage plain Hibernate Session to develop data access objects instead of binding to any Spring-specific helpers.

public class CustomerDaoImpl implements CustomerDao {

    private SessionFactory sessionFactory;

    @Autowired
    public void setSessionFactory(SessionFactory sessionFactory) {
				this.sessionFactory = sessionFactory;
    }

}

Migrating AnnotationSessionFactoryBean

In Spring ORM 3, I have often used org.springframework.orm.hibernate3.annotation.AnnotationSessionFactoryBean that subclasses LocalSessionFactoryBean and supports annotation metadata for mappings. Using AnnotationSessionFactoryBean, I can specify the packages wherein I put Hibernate annotated classes.

<bean id="sessionFactory" scope="singleton"
      class="org.springframework.orm.hibernate5.LocalSessionFactoryBean"
      p:packagesToScan="westbank.db.entity"
      p:hibernateProperties-ref="hibernateProperties" />
<bean id="hibernateProperties"
		class="org.springframework.beans.factory.config.PropertiesFactoryBean">
		<property name="properties">
			<props>
				<prop key="hibernate.connection.driver_class">${hibernate.connection.driver_class}</prop>
        ...
      </props>
	  </property>
</bean>

Since Spring 4.3, AnnotationSessionFactoryBean was deprecated in favor of Hibernate 4 and 5. Thus, LocalSessionFactoryBean must be used instead. The configuration is still similar.

LocalSessionFactoryBean localSessionBean = (LocalSessionFactoryBean) ctx.getBean("&sessionFactory");

Hibernate’s SessionFactory

The legacy Spring ORM for Hibernate 3 offers AnnotationSessionFactoryBean to initialise a sessionFactory used by other data access components.

<bean
      id="sessionFactory"
      scope="singleton"
      class="org.springframework.orm.hibernate3.annotation.AnnotationSessionFactoryBean"
      p:packagesToScan="westbank.db.entity"
      p:hibernateProperties-ref="hibernateProperties" />

When upgrading to Spring 5 / Hibernate 5, the configuration must be changed accordingly, with LocalSessionFactoryBean instead. And note that, there is no longer setting scope="singleton", otherwise the application context will fail to initialise the ‘sessionFactory’ bean.

	<bean id="sessionFactory"
		class="org.springframework.orm.hibernate5.LocalSessionFactoryBean"
		p:packagesToScan="westbank.db.entity"
		p:hibernateProperties-ref="hibernateProperties" />
  • https://stackoverflow.com/questions/18011759/hibernate-sessionfactory-vs-spring-localsessionfactorybean

Annotation Handling

Since Spring 4, DefaultAnnotationHandlerMapping has been deprecated and recommended to be replaced with RequestMappingHandlerMapping and RequestMappingHandlerAdapter. In case of legacy configuration, instead of

<bean class="org.springframework.web.servlet.mvc.annotation.DefaultAnnotationHandlerMapping" />

we should use

<bean class="org.springframework.web.servlet.mvc.method.annotation.RequestMappingHandlerMapping"/>
<bean class="org.springframework.web.servlet.mvc.method.annotation.RequestMappingHandlerAdapter"/>

Nevertheless, if we already use <mvc:annotation-driven /> , <context:component-scan base-package="..."/> and <context:annotation-config />, Spring will automatically scan and map the request handlers.
