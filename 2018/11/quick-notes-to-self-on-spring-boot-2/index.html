Quick notes to self on Spring Boot 2

Nov 21, 2018 in DEV
spring framework spring boot java convention over configuration design productivity
3 min read

I started using Spring framework back to 2009 or so to develop some demos for European projects. Most of these projects featured several Web services some share back-end databases, some interact with external services or WS-BPEL business processes, and so on.

One of the only main complaints I would have is that, configuring a complex Spring based project is truly tedious and error-prone. I have made countless trial-and errors, gone through lots of warning and errors, not to mention Maven dependency hell, the conflicts or incompatibilities among numerous dependencies. Fast forward to 2018 I want to refresh my experience through existing Spring projects. I was thinking about similar rough paths when upgrading Spring and Hibernate to some prior versions until discovering Spring Boot.

To be precise, Spring Boot is not a magic wand that would help us to navigate through the forest of Spring framework complexity or automatically resolve certain problems. It is more or less some sort of convention over configuration that might reduce significant effort on making Spring applications quickly up and running as well as ensure better compatibilities among integrated libraries.

Spring Boot brings together many essential libraries in a, as stated in the reference guide, opinionated manner. That is, some might like or dislike, especially hard core experienced developers who know almost everything about Spring. It’s natural that Spring Boot could be possibly complained about numerous design and integration decisions taken. Nevertheless, the way Spring Boot hides the underneath complexity turns out quite useful for beginners. Any inexperienced developers can start the journey with minimal effort and configuration to have a running application. After that, more and more advanced configurations or features can be gradually learned and added and/or customised until she/he does not have to rely on Spring Boot. Instead of struggling with complex settings, the developers can now better concentrate on the core business logics to be designed and implemented.

I recalled working with Spring 3 and trying to integrate Jetty HTTP server. I spent a lot of time and effort with different Jetty versions as well as other libraries to make them work together. It’s even worse when I decided to give a shot upgrading to newer Spring and Hibernate as things all turn out really messy with some mysterious errors.

Having said that, a Spring lover like me would truly appreciate the ideas and massive efforts behind the Spring Boot project to achieve an integrated set of various in-house and third-party libraries working smoothly. I think Spring development teams have made some good decisions and steered the project into right directions. With simple starting points and open to a plethora of advanced features, configurations and customisations, even experienced teams and developers would greatly benefit from that with more focused system designs and better speed of development.
